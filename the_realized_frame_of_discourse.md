# The Realized Frame of Discourse

## Abstract

Every discussion begins from an unnoticed assumption: that the discussion itself is already occurring.

A question, proof, argument, computation, simulation, plan, or observation is never suspended outside reality. It is instantiated through some organized process:

```text
neural activity
written symbols
computer memory
network communication
spoken language
institutional records
```

This observation is more fundamental than questions about whether the subject of discussion is physical, mathematical, fictional, or abstract.

Before asking what **x** is, there must already exist a realized representation through which **x** participates in the discussion.

The central claim of this framework is therefore:

> Every discourse has a realized frame.

This is not a claim about the ontology of the referent.

It is a claim about the ontology of the discussion.

---

# 1. The Realized Frame of Discourse

Suppose we ask:

```text
Do unicorns exist?
```

The question itself is already realized.

The symbols, sounds, memories, or neural organizations participating in the discussion exist as organized processes.

Only afterwards do we investigate the referent.

Thus we distinguish:

```text
realized representation
```

from

```text
referent
```

The first is required for discourse.

The second remains open.

---

# 2. Representation Comes Before Ontology

Every inquiry implicitly follows:

```text
realized process
        ->
representation of x
        ->
discussion about x
        ->
possible conclusions about x
```

Not

```text
x
        ->
discussion
```

The framework therefore does not assume that every referent exists physically.

It assumes only that every discussion is instantiated.

---

# 3. Instantiation Is Not Grounding

The framework does not require metaphysical grounding.

Instead it requires an instantiation context.

For example, the Latin alphabet may participate in:

```text
English
German
C++
Python
mathematics
random text
```

The alphabet is not grounded in any one of these.

Rather, different organizational systems interpret the same instantiated symbols differently.

Meaning depends upon participation in an organized system, not exclusive attachment to one domain.

---

# 4. Organizational Context

Every representation participates inside some organization.

Examples include:

```text
human language
programming language
compiler
mathematical proof
legal institution
scientific measurement
```

The same physical symbols may therefore serve different purposes while remaining the same instantiated symbols.

---

# 5. Abstraction Preserves Organization

Abstraction does not merely omit detail.

It preserves selected organization while suppressing the rest.

Compression therefore has no intrinsic stopping point.

One may write:

```text
sort(array)
```

or

```text
do_it()
```

or

```text
.
```

or produce an empty file.

Each is syntactically possible.

The important distinction is not how much was omitted.

It is how much organization remains available.

Thus:

> Abstraction is limited not by syntax, but by preservation.

---

# 6. Capability Depends on Preserved Organization

An empty file is still a file.

A random string is still text.

Neither necessarily preserves the organization required to execute a sorting algorithm.

Therefore:

```text
syntactic existence
!=
retained capability
```

Capability depends upon preserved organizational relations.

---

# 7. Intermediate Organization

Any executable process requires organization sufficient for continuation.

A planner may omit intermediate detail.

Reality does not.

Whether constructing software, buildings, proofs, or biological organisms, each transition must preserve or produce the organization needed for the next transition.

The omitted steps remain omitted only from description.

Not from execution.

---

# 8. Interfaces Are Organizational Certificates

A high-level operation such as

```text
sort(array)
```

does not contain its implementation.

Instead it certifies that sufficient organization exists somewhere to resolve the request.

Thus an interface is not magical compression.

It is an access point into retained organization.

---

# 9. The Principle of Organizational Preservation

The meaningful measure of abstraction is therefore not information removed.

It is organization preserved.

The same representation may preserve:

```text
syntax
semantics
interfaces
proof obligations
connectivity
executability
```

depending upon purpose.

Different purposes require different preserved relations.

---

# 10. The Central Principle

The framework may be summarized as follows.

> Before discussing what something is, the discussion itself is already instantiated.

Everything else follows from this observation.

Representations, abstractions, proofs, plans, algorithms, theories, and concepts do not float outside reality while they are being used.

They participate through realized organizations.

The remaining question is never merely whether information exists.

The question is:

> Which organization has been preserved, and is it sufficient for the intended purpose?

This shifts attention away from metaphysical debates about abstraction itself toward the operational conditions that make abstraction usable.

Abstractions are therefore best understood not as escapes from realization, but as organized reductions that preserve the relations required for a task.
